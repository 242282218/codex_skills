# 🛸 技能：颠覆式创新分析 (Revolution Analysis)

## 使用场景

> 当用户要求提出功能上的颠覆性创新建议，或者跳出常规项目管理视角、寻求"突破"、"好玩"或"未来感"的产品 Feature 时激活此 skill。该技能强制进入"创新部门 / 创意总监"视角，拒绝平庸的修补策略（如基础状态维护、UI 细调、常规 bug 修复）。
>
> **激活前必须确认 `required_context` 中的四项上下文已提供。** 若用户未提供，应主动询问后再推演，以确保颠覆方向不发生偏移。

---

## ⚠️ 模型知识库免责声明

> 本技能的颠覆质量高度依赖**当前时效性信息**。所有模型内置知识均有训练截止日期，可能严重滞后于当前行业现实。
>
> 因此，**必须强制执行 Step 0（联网情报侦察）**，以当前真实的技术格局、竞品动态和最新研究成果作为推演燃料。跳过 Step 0 的推演结果视为无效。

---

## 执行步骤

---

### 🔍 Step 0 — 联网情报侦察（强制，不得跳过）

> **目的**：用最新的真实世界信息替代模型内置的陈旧知识，防止颠覆脑暴建立在过时的行业假设上。
>
> **要求**：必须实际调用 `search_web` 工具，严禁凭记忆跳过搜索。每一批搜索完成后，先记录关键发现，再进入下一批。

#### 第一批：定位当前产品所在赛道的最新格局

针对用户项目所属领域，并行执行以下搜索（至少完成其中 4 条）：

```
[领域] "2025" OR "2026" latest trends
[领域] disruption startup product hunt
[领域] [目标用户] pain points reddit
[产品核心功能] alternative app OR tool 2025
[领域] open source innovative project github
```

> 记录发现：当前赛道有哪些尚未被主流产品解决的痛点？有哪些新兴玩家正在改变规则？

---

#### 第二批：跨界参照系搜索（为维度一做弹药）

从以下 **3~4 个无关行业** 中搜索最具创意的近期产品机制，每个行业执行 1~2 条搜索：

```
"game mechanic" "2025" innovative UX engagement
crypto NFT "social layer" user retention mechanism
fintech "micro-interaction" OR "gamification" 2024 2025
military intelligence tool UX "situation awareness"
spatial computing "Vision Pro" app interaction pattern
```

> 记录发现：这些领域有哪些交互机制或商业逻辑可以被"强行移植"到当前项目？

---

#### 第三批：未来技术边界搜索（为维度四做燃料）

```
"brain computer interface" consumer app 2025 2026
AI agent "ambient computing" UX concept 2025
"spatial computing" developer tools 2025
LLM "real-time" multimodal interaction pattern
"AI companion" relationship model product 2025
```

> 记录发现：哪些技术已经从科幻进入早期落地阶段？这些技术假设是否可用于未来推演中？

---

#### 第四批：极客用户与地下文化侦察（为维度二 Pro 极客模式做弹药）

```
[产品领域] "power user" OR "hardcore" OR "geek" community
[产品领域] subreddit OR hacker news discussion 2024 2025
[产品领域] "hidden feature" OR "undocumented" OR "API hack"
[竞品名] "I wish it could" OR "why doesn't it" site:reddit.com
```

> 记录发现：极客用户在哪些主流产品没满足的隐性需求上抓狂？他们的黑客式用法揭示了什么产品空白？

---

#### 第五批：游戏化前沿机制搜索（为维度三做弹药）

```
"octalysis" "2024" OR "2025" case study product
gamification "flow state" design pattern app 2025
"social proof" OR "FOMO" mechanic app engagement 2025
"intrinsic motivation" UX design pattern 2025
viral loop product design mechanism 2025
```

> 记录发现：哪些游戏化机制已在其他产品中被证明有效？有没有反常识的"反游戏化"趋势正在兴起？

---

#### 情报汇总（进入 Step 1 前必须输出）

完成以上五批搜索后，必须先输出一个简短的**情报摘要**，格式如下：

```markdown
## 🔍 情报摘要（Step 0 产出）

**赛道格局关键发现**：[3~5 条最有价值的行业洞察]

**可移植的跨界机制**：[2~3 个可以被强行借鉴的异业逻辑]

**近期可用的技术假设**：[2~3 个已进入早期落地的前沿技术]

**极客用户未被满足的隐性需求**：[2~3 个从极客社区发掘的痛点]

**高效游戏化机制参考**：[2~3 个经过验证的非传统游戏化机制]

**最大的惊喜发现**：[1 条在搜索过程中完全没预料到的信息]
```

> ⚠️ 如果某条搜索没有返回有价值的信息，必须**换词重搜**，不得以"无结果"为由跳过该批次。

---

### Step 1 — 探索发散起点

- 结合 Step 0 情报摘要中的发现，提炼当前项目的核心用户价值。
- 输出一句"本项目的本质是：___"，并**注明这个定义是否因 Step 0 的信息而被修正**（如果是，说明修正前后的差异）。

---

### Step 2 — 强制执行四大维度脑暴推演

> **注意**：四大维度的推演必须**显式引用 Step 0 情报摘要中的至少一条发现**，不得凭模型内置知识空想。

#### 🌀 维度一：跨界脑洞碰撞 (Cross-pollination)

- 使用 **SCAMPER 创新法则**，将当前项目与 **Step 0 第二批搜索中发现的跨界参照系** 强制结合（至少 3 个行业）。
- 产出 **3~5 个** 具有异业逻辑的前沿 Feature。
- 每个 Feature 须标注：「借鉴自 [行业] 的 [具体机制]，来源：Step 0 情报」。

#### 🪃 维度二：痛点逆向工程 (Pain-point Reverse-Engineering)

按以下结构强制推演，不得跳过任何一步：

1. **淘汰假设**：结合 Step 0 第一批发现的"新兴替代者"，如果明天这个工具被其取代，会是什么形态？
2. **替代形态描述**：具体描绘这个新形态的交互方式与价值主张。
3. **极客对照矩阵**（结合 Step 0 第四批极客社区发现填充）：

| 维度 | 普通用户（98%） | 硬核极客用户（1%） |
|------|---------------|-----------------|
| 核心需求 | … | … |
| 最大痛点 | … | … |
| 愿意为之付费的功能 | … | … |

4. **Pro 极客彩蛋设计**：至少 2 个具体功能，**必须基于 Step 0 第四批社区洞察**，而非凭空想象。

#### 🕹️ 维度三：游戏化与心流设计 (Gamification & Engagement)

- 强制应用 **八角行为学 (Octalysis)** 框架，至少覆盖其中 3 个核心驱动力。
- **必须引用 Step 0 第五批**中发现的至少 1 个经验证的非传统游戏化机制作为参考基准。
- 设计 **3 个** 高频心流交互或动态激励反馈机制。
- **严禁** 积分签到、打卡、徽章收集等陈词滥调。
- 每个机制须说明触发场景、即时反馈形式、自传播行为。

#### 🔭 维度四：未来场景推演 (Future Scenario Generation)

- **必须基于 Step 0 第三批**发现的"已进入早期落地阶段的技术"来选择技术假设，而非使用陈旧的科幻设定。
- 产出 **3 个** 未来世代概念原型，每个使用以下"概念原型卡片"格式：

```
【概念原型卡片】
名称：___
技术假设：___（来自 Step 0 第三批的哪项技术发现）
交互方式：___（用户如何与之互动，越具体越好）
颠覆点：___（它改变了用户与产品之间的哪种关系模型？）
与当前技术栈的距离：___（近期 / 中期 / 远期）
```

---

### Step 3 — 自检：颠覆烈度测试

> 在进入报告撰写之前，做一次强制自检：
>
> ✅ 如果这份报告放到一个传统产品经理评审会上，它是否会让保守派感到**不舒服**？
> ✅ 所有想法的本质是否改变了**用户与产品之间的关系模型**，而不仅仅是"做得更好/更快"？
> ✅ 报告中有多少内容实际引用了 Step 0 的搜索发现？（少于 5 次引用视为情报利用不足，需补搜）
>
> 如果以上任意一项未通过，必须重新推演至少一个维度，不得跳过。

---

### Step 4 — 撰写并提交《颠覆式创意报告》

- 将推演结果整理为结构清晰的 Markdown 报告，可直接在对话中输出，也可写入 `docs/plans/YYYY-MM-DD-revolution-ideas.md` 作为备忘录。
- **【强制要求】** 报告最后一节必须包含 **落地建议 (Actionable Next Steps)**，使用以下 MVI 模板提取 **1~2 个** 最小可行性创新：

```markdown
### MVI-01：[创新点名称]
- **来源维度**：[跨界碰撞 / 痛点逆向 / 游戏化 / 未来推演]
- **情报依据**：[Step 0 中哪条搜索发现支撑了这个 MVI]
- **核心颠覆逻辑**：[一句话说明为何是非渐进式创新，而非功能增强]
- **当前技术栈可行性**：[具体说明可利用哪些现有模块/API/库实现]
- **预计实现复杂度**：[低 / 中 / 高]
- **进入编码条件**：用户须明确说"帮我做这个"才可切换至具体编码阶段
```

---

## ⛔ 雷区 (Anti-Pattern)

### 低级雷区（直接判定推演失败）
- **跳过 Step 0 联网搜索** — 绝对禁止，哪怕用户没有明说"要搜索"也必须执行。
- **"我来帮你优化性能吧"** — 坚决不可以。
- 出现"改善UI排版"、"加个设置页"、"提升运行效率"等渐进式操作 → **推演失败，重来**。

### 伪颠覆陷阱（更隐蔽，同样致命）
> 如果一个想法的本质是"把现有功能做得更好/更快/更准"，即使包裹着"AI自动化"、"大模型赋能"的外衣，也属于伪颠覆。
>
> **真正的颠覆必须改变用户与产品之间的关系模型，或产品的价值交付方式本身。**

### 搜索敷衍雷区
> - 只搜索 1~2 条就进入脑暴 → 情报不足，结论不可信。
> - 用模型内置知识"伪造"搜索结果 → 禁止。
> - 搜索结果与推演内容之间没有可追溯的引用关系 → 视为未搜索。

---

## 🎯 语调 (Tone)

- 充满极客感、锐利、一针见血，带有实验性质的狂热。
- 不写企业报告体，不用"我们可以考虑"、"建议适当引入"等保守措辞。
- 每一个 Feature 名称都应该让人想立刻转发给朋友。

---

## 🚫 不包含实现代码 (No Implementation Yet)

本阶段纯属"研发创新部门"，只提供点子、MVI 与产品方向说明。**不要自作主张直接去改源代码！** 用户必须先对某个点子明确表示："这个很棒，帮我写实现代码"，才能切换至具体的编码技能。
